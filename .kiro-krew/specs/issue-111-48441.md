# Design Specification: Consolidate About Dialog Rendering and Fix Re-rendering

Closes #111

## Solution Approach

The current about dialog implementation has two architectural problems:

1. **Code Duplication**: Identical content structure is built in two places:
   - `internal/tui/commands.go:handleAbout()` - Initial render with placeholder "Checking for updates..."
   - `internal/tui/tui.go:updateCheckMsg` handler - Complete re-render with actual update status

2. **Inefficient Re-rendering**: The entire about dialog is rebuilt just to update the status line

**Solution**: Create a centralized about dialog content builder and implement partial content updates for efficient re-rendering.

## Implementation Notes

### Code Efficiency
- Minimal struct - only store necessary state for content building
- Reuse base content array, only rebuild status section
- Thread-safe not required (single TUI goroutine)

### Backwards Compatibility  
- All existing keyboard shortcuts and behavior preserved
- Same visual appearance and overlay styling
- Identical error handling and edge cases

### Error Handling
- Graceful fallback if about dialog initialization fails
- Preserve existing error handling for version info and update checks
- No new error states introduced

This design eliminates code duplication while improving performance through targeted partial updates, maintaining full compatibility with existing functionality.
