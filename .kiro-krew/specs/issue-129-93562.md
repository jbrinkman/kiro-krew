# Design Specification: Hide Update Status When No Releases Exist

**Issue:** #129  
**Title:** Hide update status in about dialog when no releases exist  
**Closes:** #129

## Problem Analysis

The REPL TUI about dialog currently shows an error message when checking for updates because no GitHub releases exist yet:

```
Update Status: Unable to check for updates
  Error: gh release view failed: exit status 1
```

**Root Cause:** The `GetLatestRelease` function in `internal/github/client.go` treats all `gh release view` failures identically, but the "no releases exist" scenario (exit code 1 with "release not found") should be handled differently from genuine errors.

**Current Flow:**
1. `handleAbout()` calls `checkForUpdateCmd()` 
2. `checkForUpdateCmd()` calls `github.GetLatestRelease()`
3. `GetLatestRelease()` executes `gh release view` command
4. When no releases exist, command returns exit code 1 
5. Function returns generic error wrapping the command failure
6. TUI displays error in about dialog

## Solution Approach

**Strategy:** Distinguish "no releases exist" from other command failures by examining both exit code and stderr content.

**Key Insight:** When no releases exist, `gh release view` specifically returns:
- Exit code: 1
- Stderr: "release not found"

Other failures (network, auth, etc.) will have different stderr messages.

## Architecture Changes

### 1. Enhanced Error Handling in GitHub Client

**File:** `internal/github/client.go`

Add a new error type to distinguish "no releases" from other failures:

```go
// ErrNoReleases indicates no releases exist for the repository
var ErrNoReleases = errors.New("no releases found")
```

**Modified `GetLatestRelease` function:**
- Capture both stdout and stderr from `gh release view` 
- Check if exit error with code 1 AND stderr contains "release not found"
- Return `ErrNoReleases` for this specific case
- Return wrapped generic error for all other failures

### 2. Conditional Update Status Display

**File:** `internal/tui/tui.go`

**Modified `updateCheckMsg` handling:**
- Check if error is `ErrNoReleases` using `errors.Is()`
- When `ErrNoReleases`: call `UpdateStatusLine` with empty slice to hide section
- For other errors: preserve existing error display behavior
- For successful responses: preserve existing success display behavior

## Relevant Files

1. **`internal/github/client.go`** - Modify `GetLatestRelease()` function for enhanced error detection
2. **`internal/tui/tui.go`** - Update `updateCheckMsg` case to handle `ErrNoReleases`
3. **`internal/tui/about.go`** - Ensure `UpdateStatusLine` properly handles empty status (already supports this)

## Step-by-Step Implementation

### Step 1: Enhance GitHub Client Error Detection
- **Location:** `internal/github/client.go`
- **Task:** Add `ErrNoReleases` error variable
- **Task:** Modify `GetLatestRelease()` to:
  - Use `cmd.CombinedOutput()` instead of `cmd.Output()` to capture stderr
  - Check for exit code 1 AND stderr containing "release not found"
  - Return `ErrNoReleases` for no-releases case
  - Return wrapped generic error for other failures
- **Acceptance Criteria:** 
  - When no releases exist, function returns `ErrNoReleases`
  - Other failures continue to return descriptive error messages
  - Successful releases continue to work normally

### Step 2: Update TUI Message Handling  
- **Location:** `internal/tui/tui.go` 
- **Task:** Modify `updateCheckMsg` case to:
  - Import `errors` package if not already imported
  - Check `errors.Is(msg.err, github.ErrNoReleases)`
  - Call `UpdateStatusLine([]string{})` for no-releases case to hide section
  - Preserve existing logic for other errors and success cases
- **Acceptance Criteria:**
  - About dialog shows no update status section when no releases exist
  - Error messages still appear for network/auth/other failures
  - Success messages continue to work when releases do exist

### Step 3: Verification Testing
- **Task:** Test behavior with projects that have no releases
- **Task:** Test behavior with projects that have releases  
- **Task:** Test behavior with network failures
- **Acceptance Criteria:**
  - Clean about dialog (no update section) for repos without releases
  - Normal update checking for repos with releases
  - Error messages for genuine failures (network, auth, etc.)
