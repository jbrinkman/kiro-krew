# Design Specification: Implement Semantic Releases with Automated Changelog Generation

**Issue**: #143
**Title**: Implement semantic releases with automated changelog generation  
**Closes**: #143

## Solution Approach

Replace the current manual/auto-increment versioning system with semantic-release that automatically calculates version numbers based on conventional commit messages and generates comprehensive changelogs. This approach leverages the project's existing conventional commit standards while maintaining backward compatibility with current build processes.

### Key Architectural Decisions

1. **Use semantic-release/semantic-release** as the primary tool for version calculation and release automation
2. **Preserve existing workflow_dispatch trigger** to maintain manual release capability
3. **Update version.json atomically** to ensure consistent version state across all build processes
4. **Generate CHANGELOG.md** at project root using semantic-release plugins
5. **Graceful no-release handling** when no version-impacting commits are found

## Relevant Files

### Files to Modify
- `.github/workflows/release.yml` - Replace version determination logic with semantic-release
- `package.json` (NEW) - Required by semantic-release for configuration and dependency management
- `.releaserc.json` (NEW) - Semantic-release configuration file

### Files Referenced/Used
- `internal/version/version.json` - Updated by semantic-release via custom script
- `Taskfile.yml` - Continue using existing `version:set` and `build:release` tasks
- `CHANGELOG.md` (NEW) - Generated and maintained by semantic-release

### Files Not Changed

- All Taskfile.yml tasks - Preserved as-is for backward compatibility
- CI workflow (`.github/workflows/ci.yml`) - No changes needed

## Team Orchestration

### Version Calculation Flow
1. **semantic-release** analyzes commits since last release using conventional commit patterns
2. **Version calculation** follows semantic versioning rules:
   - `BREAKING CHANGE` or `feat!`/`fix!` → major version bump
   - `feat:` commits → minor version bump
   - `fix:` commits → patch version bump
   - Other types (`docs`, `chore`, etc.) → no version bump
3. **Early exit** if no version-impacting changes found (prevents unnecessary releases)

### Release Artifact Generation Flow
1. **semantic-release** determines new version and updates `internal/version/version.json`
2. **Task runner** builds release artifacts using existing `task build:release`
3. **GitHub release** created with generated changelog content
4. **CHANGELOG.md** updated and committed back to repository

## Step-by-Step Task Breakdown

### Task 1: Configure semantic-release Dependencies
**Acceptance Criteria**: 
- Create `package.json` with semantic-release and required plugins
- Configure Node.js runtime in GitHub Actions workflow
- Ensure semantic-release executable is available

**Implementation**:
- Add minimal `package.json` with semantic-release dependencies
- Update workflow to install Node.js and npm dependencies
- Use `npx semantic-release` for execution

### Task 2: Create semantic-release Configuration
**Acceptance Criteria**:
- Configure `.releaserc.json` with appropriate plugins for Go projects
- Enable changelog generation with conventional-changelog plugin  
- Configure custom exec plugin for updating version.json
- Set up GitHub releases with changelog content

**Implementation**:
- Use `@semantic-release/commit-analyzer` for version calculation
- Use `@semantic-release/release-notes-generator` for changelog generation
- Use `@semantic-release/changelog` for CHANGELOG.md maintenance
- Use `@semantic-release/exec` for version.json updates via Task
- Use `@semantic-release/github` for GitHub releases and asset uploads

### Task 3: Modify Release Workflow
**Acceptance Criteria**:
- Replace manual version determination with semantic-release
- Preserve workflow_dispatch trigger with optional version override
- Maintain existing build and asset upload processes
- Handle no-release scenarios gracefully

**Implementation**:
- Add Node.js setup step before semantic-release execution
- Run semantic-release in dry-run mode first to determine if release is needed
- Execute full semantic-release if version bump detected
- Skip release creation if no version-impacting changes found

### Task 4: Version.json Integration
**Acceptance Criteria**:
- semantic-release updates `internal/version/version.json` with calculated version
- Preserve existing JSON structure and fields
- Maintain compatibility with Go embed and version loading

**Implementation**:
- Configure semantic-release exec plugin to call `task version:set VERSION=${nextRelease.version}`
- Ensure version.json retains all existing fields (version, prerelease, commit_hash, build_timestamp)
- Update commit_hash and build_timestamp during release process

### Task 5: Changelog Generation  
**Acceptance Criteria**:
- Generate CHANGELOG.md with categorized release notes
- Include Features, Bug Fixes, Breaking Changes sections
- Follow conventional changelog formatting standards
- Integrate changelog content into GitHub release notes

**Implementation**:
- Configure conventional-changelog preset for standardized formatting
- Group commits by type (feat, fix, breaking changes, etc.)
- Include commit hashes and PR references where applicable
- Commit generated CHANGELOG.md back to repository

### Task 6: Manual Override Support
**Acceptance Criteria**:
- Preserve ability to specify version manually via workflow_dispatch
- Manual version overrides semantic-release calculation
- Manual releases still update version.json and generate changelog

**Implementation**:
- Conditional logic in workflow to check for manual version input
- Skip semantic-release version calculation if manual version provided
- Use manual version for Task version:set and release creation

### Task 7: Backward Compatibility Testing
**Acceptance Criteria**:
- All existing Taskfile.yml tasks continue working
- Binary version metadata injection still functions
- Build processes remain unchanged
- Version loading in Go code works as before

**Implementation**:
- Test `task build` with new version.json format
- Verify ldflags version injection during compilation  
- Confirm `internal/version` package loads version correctly
- Validate cross-platform build artifacts

## Validation Commands

### Semantic Release Dry Run
```bash
npx semantic-release --dry-run
```
**Expected**: Analyze commits and determine next version without creating release

### Manual Release Test
```bash
gh workflow run release.yml -f version="1.2.3"
```
**Expected**: Create release with specified version, update version.json, generate changelog

### Automatic Release Test
```bash
gh workflow run release.yml
```  
**Expected**: Analyze commits, calculate semantic version, create release if version bump warranted

### Version Integration Test
```bash
task build && ./dist/kiro-krew --version
```
**Expected**: Display semantic version from updated version.json

### Changelog Validation
```bash
cat CHANGELOG.md
```
**Expected**: Properly formatted changelog with release notes categorized by commit type

### No-Release Scenario Test  
**Setup**: Ensure HEAD contains only non-version-impacting commits (docs, chore, etc.)
```bash
npx semantic-release --dry-run
```
**Expected**: Output indicating no release needed, exit code 0

### Build Compatibility Test
```bash
task build:release
ls -la dist/release/
```
**Expected**: All platform-specific binaries built successfully with embedded version metadata

### Integration Points
- GitHub Actions workflow for automated execution
- Task runner for version management and build processes
- Git repository for changelog commits and release tags

## Risk Mitigation

### Breaking Changes Prevention
- Preserve all existing Taskfile.yml task interfaces
- Maintain version.json structure and field names
- Keep build process and ldflags injection unchanged

### Rollback Strategy  
- Manual version specification capability preserved as fallback
- Existing version:set task continues working independently
- Git history and tags provide audit trail for version changes

### Testing Strategy
- Dry-run testing before production deployment
- Backward compatibility validation for all build processes
- Manual override testing for emergency release scenarios
