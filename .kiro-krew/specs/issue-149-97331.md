# Design Specification: Early Docker Availability Check for Sandbox Mode

**Issue:** #149 - Check Docker availability early in evaluation startup when using --sandbox

Closes #149

## Problem Statement

Currently, when running `kiro-krew eval architect --sandbox`, Docker availability is only checked during individual container creation in `sandbox.NewContainer()`. This results in poor user experience where multiple test cases fail individually instead of providing one clear early error message.

Users waste time watching multiple failed test executions when Docker is unavailable, rather than getting immediate feedback at startup.

## Solution Approach

Implement early Docker availability checking at the evaluation startup phase in `internal/eval/runner.go` when sandboxing is enabled. This will:

1. **Fast-fail behavior**: Check Docker immediately when `--sandbox` flag is detected
2. **Reuse existing logic**: Leverage the proven Docker check from `sandbox.NewContainer()`
3. **Minimal disruption**: Only add checks where sandboxing is requested
4. **Consistent error messaging**: Use existing Docker error format for familiarity

## Architecture Strategy

### Current Flow
```
RunWithOptions() → createContainerConfig() → Run()/runSingleTestCase() → evaluate() → invokeAgentInContainer() → sandbox.NewContainer() → Docker check fails
```

### Proposed Flow
```
RunWithOptions() → [Early Docker Check if --sandbox] → createContainerConfig() → Run()/runSingleTestCase() → ... (continue normally)
```

### Early Check Integration Points
- `RunWithOptions()` function: Before performance profiling starts
- `Run()` function: After rubrics validation, before progressive evaluation
- Only when `options.Sandbox && !options.NoSandbox` condition is met

## Relevant Files

### Primary Implementation
- **`internal/eval/runner.go`** - Main implementation file

  - Integrate check into `RunWithOptions()` and `Run()` 
  - Import Docker client packages

### Reference Implementation  
- **`internal/eval/sandbox/container.go`** - Source for Docker check logic
  - Extract reusable Docker ping logic from `NewContainer()`
  - Maintain existing timeout (5 seconds) and error message format

### Testing Support
- **`internal/eval/sandbox/container_test.go`** - Reference for test patterns
  - Use similar `skipIfNoDocker` pattern for new tests

## Step-by-Step Task Breakdown

### Task 1: Extract Docker Check Function
- **Description**: Create reusable Docker availability check function
- **Location**: `internal/eval/runner.go`
- **Implementation**:
  - Add `checkDockerAvailability() error` function
  - Use Docker client creation and ping logic from `sandbox.NewContainer()`
  - 5-second timeout with context cancellation
  - Return descriptive error matching existing format
- **Acceptance Criteria**:
  - Function returns `nil` when Docker is available
  - Function returns descriptive error when Docker daemon not running
  - Function returns descriptive error when Docker client cannot be created
  - Function completes within 5 seconds maximum

### Task 2: Integrate Check into RunWithOptions()
- **Description**: Add early Docker check in main evaluation entry point
- **Location**: `internal/eval/runner.go` - `RunWithOptions()` function
- **Implementation**:
  - Add check after options parsing, before performance profiling starts
  - Only execute when `options.Sandbox && !options.NoSandbox`
  - Return immediately on Docker unavailability
- **Acceptance Criteria**:
  - Check only runs when `--sandbox` flag is used
  - Check does not run when `--no-sandbox` flag is used  
  - Check does not run when neither flag is specified
  - Evaluation exits immediately with clear error when Docker unavailable

### Task 3: Integrate Check into Run() Function
- **Description**: Add early Docker check in legacy Run() function
- **Location**: `internal/eval/runner.go` - `Run()` function  
- **Implementation**:
  - Add check after rubrics validation, before creating results directory
  - Only execute when `cConfig != nil` (container config provided)
  - Maintain existing function signature
- **Acceptance Criteria**:
  - Check only runs when container configuration is provided
  - Check integrates cleanly with existing startup sequence
  - Evaluation exits immediately with clear error when Docker unavailable

### Task 4: Add Required Imports
- **Description**: Add Docker client imports to runner.go
- **Location**: `internal/eval/runner.go` - import section
- **Implementation**:
  - Add `"github.com/docker/docker/client"`
  - Add `"context"`
  - Add `"time"` (if not already present)
- **Acceptance Criteria**:
  - Code compiles successfully
  - No import conflicts
  - Follows existing import organization pattern

### Task 5: Maintain Error Message Consistency
- **Description**: Ensure new Docker errors match existing format
- **Location**: `internal/eval/runner.go` - `checkDockerAvailability()` function
- **Implementation**:
  - Use format: "Docker is not running. Start Docker and try again: %w"
  - Preserve error wrapping for detailed diagnostics
  - Match error handling patterns from `sandbox.NewContainer()`
- **Acceptance Criteria**:
  - Error messages are identical to existing Docker errors
  - Users get familiar, actionable error messages
  - Error context is preserved for debugging

## Implementation Notes

### Minimal Code Changes
- Reuse existing Docker client initialization patterns
- Leverage proven timeout and error handling logic
- Maintain backward compatibility with all existing functionality

### Error Handling Strategy
- Use consistent error wrapping with `fmt.Errorf`
- Preserve original Docker errors for detailed diagnosis
- Exit immediately without partial execution attempts

### Performance Considerations  
- Single 5-second check vs. multiple per-container failures
- Check only runs when explicitly requested via `--sandbox`
- No impact on non-sandboxed evaluation performance

### Testing Strategy
- Unit tests for `checkDockerAvailability()` function
- Integration tests covering all flag combinations
- Error path testing with mocked Docker unavailability
- Performance regression testing for check overhead

This design ensures fast failure with clear user feedback while maintaining full compatibility with existing evaluation workflows.
