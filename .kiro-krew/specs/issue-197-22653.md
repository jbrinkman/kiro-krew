# Design Specification: Prevent Phase-Based Implementation Breakdowns in Architect

**Issue**: #197 - Architect should not break down work into phases - requires single PR implementation

**Closes**: #197

## Solution Approach

Update the architect agent prompt to explicitly prohibit phase-based task breakdowns and emphasize unified, single-PR implementation planning. The core issue is that the architect currently creates specifications that suggest incremental delivery across multiple PRs, but kiro-krew's workflow orchestrates each issue as a complete solution delivered via a single pull request.

Key changes needed:
1. **Explicit Phase Prohibition**: Add clear language prohibiting "Phase 1", "Phase 2", etc. breakdowns
2. **Single-PR Emphasis**: Reinforce that all tasks must contribute to complete issue resolution in one PR
3. **Implementation Order Focus**: Reframe task breakdowns as logical implementation sequences within a single development cycle
4. **Builder Context Clarity**: Clarify that kiro-krew spawns one builder agent per issue, not multiple builders for phases

The solution maintains all existing architect functionality while preventing problematic phase-based planning that conflicts with the single-PR workflow.

## Implementation Details

### Key Prompt Additions

**Implementation Approach Section**:
```markdown
## Implementation Approach

**CRITICAL**: Kiro-krew processes each issue as a single, complete solution delivered via one pull request. Do NOT break work into phases, incremental delivery, or multi-PR approaches.

- Task breakdowns represent **logical implementation order** within a single development cycle
- All tasks must contribute to **complete issue resolution** in one PR
- Kiro-krew spawns **one builder agent** per issue that executes tasks sequentially
- Design specifications provide implementation roadmaps, not multi-phase project plans

**Prohibited Patterns**:
- "Phase 1: Foundation", "Phase 2: Core Implementation"
- Incremental delivery suggestions
- Multi-PR implementation plans
- Partial completion milestones

**Required Approach**:
- Sequential task organization for single implementation cycle
- Complete feature/fix delivery in one pull request
- All acceptance criteria achievable through single builder execution
```

**Enhanced Task Breakdown Language**:
```markdown
### Task N: [Implementation Step Description]
**Acceptance Criteria**:
- [Specific, complete requirements]
- [All criteria must be achievable in single PR]
- [Focus on implementation completion, not phases]
```

## Implementation Notes

**Backward Compatibility**: The updated prompt preserves all existing architect functionality while adding phase-prohibition constraints. Existing specifications in `.kiro-krew/specs/` demonstrate proper single-PR planning patterns.

**Builder Integration**: The architect's task breakdown directly feeds the builder agent, which executes "ONE task at a time" within the single development cycle. This natural workflow should not be disrupted by phase-based thinking.

**Quality Assurance**: The updated approach ensures that all architect specifications support the validator's requirement for complete, testable implementations rather than partial deliveries.

**Workflow Consistency**: The change aligns architect planning with krew-lead orchestration, which creates one worktree, one branch, and one PR per issue regardless of complexity.
