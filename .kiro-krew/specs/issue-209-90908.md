# Design Specification: TUI Footer Misrendered on Planning Tab

**Issue:** #209 - TUI Footer Misrendered on Planning Tab - Footer Rendering Bug  
**Closes #209**

## Problem Analysis

The TUI footer section displays missing footer elements (input box, status bar, and separator) when the planning tab is active. The root cause is an architectural mismatch in the view rendering system where planning tabs bypass the footer rendering entirely.

### Root Cause Investigation

**Current Flow Analysis:**

1. **Main Tab (Working):**
   ```go
   content = renderBaseView()  // Includes footer via footerManager.RenderDropdownWithFooter()
   ```

2. **Planning Tab (Broken):**
   ```go
   content = activeTab.View()  // Planning tab's View() method - NO footer included
   ```

**The Issue:** Planning tabs render their content through `PlanningTab.View()` which only returns the chat interface without any footer. The footer system (`FooterManager`) is only invoked in `renderBaseView()`, which is only used for the main tab.

**Expected vs Actual:**
- **Expected:** All tabs should display a consistent footer with input box, status bar, and separator
- **Actual:** Planning tabs show only the chat interface with no footer elements

## Solution Approach

Implement a **unified tab rendering system** that ensures all tab types include proper footer rendering while maintaining tab-specific content areas.

### Architecture Decision

Create a consistent rendering pattern where:
1. All tabs get a footer (input box + status bar + separator)  
2. Each tab type renders its content area above the footer
3. The footer shows context-appropriate information based on active tab type
4. Footer input remains functional across all tab types
