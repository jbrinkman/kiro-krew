# Issue #212 - Planning Tab Border Removal Design Specification

Closes #212

## Solution Approach

Remove all border rendering from the planning tab content area to maximize space usage while maintaining the clean two-row footer layout established in Issue #195. The solution focuses on eliminating visual borders that consume horizontal space (-2 width adjustments) while preserving all functionality and visual separation through the existing footer system.

The architecture leverages the unified rendering system where:
- Tab content area uses full available width
- Footer system provides visual separation
- No redundant visual boundaries within content area
- Responsive design maintained across terminal widths

## Relevant Files

### Files to Modify:
- `internal/tui/planning_tab.go` - Remove border styling from View() method, update width calculations
- `internal/tui/styles.go` - Update GetPlanningBorderStyle method or remove if no longer needed

### Files Referenced:
- `internal/tui/footer.go` - Footer system that provides visual separation (no changes needed)
- `internal/tui/tab_manager.go` - Tab content area height calculation system (no changes needed)
- `internal/tui/tui.go` - Unified rendering system integration (no changes needed)

## Team Orchestration

This is a single-developer implementation that removes visual elements rather than adding complexity. The work can be completed in one cohesive implementation cycle with the following logical sequence:

1. **Planning Tab View Logic** - Remove border styling and width calculations
2. **Styling System Updates** - Update or remove unused border style methods
3. **Verification** - Ensure all functionality remains intact and responsive design works

No parallel work opportunities exist as the changes are tightly coupled within the planning tab rendering system.

## Step-by-Step Task Breakdown

### Task 1: Remove Border Styling from Planning Tab View Method
**Acceptance Criteria**:
- Remove `borderStyle := pt.styles.GetPlanningBorderStyle(pt.width)` usage from View() method
- Remove `messageArea := borderStyle.Width(pt.width - 2).Height(messageHeight).Render(pt.viewport.View())` container
- Render `pt.viewport.View()` directly without border container wrapping
- Update viewport width calculation to use full available width (remove -2 adjustments)
- Update responsive width calculations for narrow terminals (remove -1 adjustments for borders)
- Ensure messageArea renders clean content without border styling
- Maintain existing separator and inputArea rendering unchanged

**Dependencies**: None

### Task 2: Update Width Calculations for Full Space Usage
**Acceptance Criteria**:
- Change `viewportWidth := pt.width - 2` to `viewportWidth := pt.width`
- Remove narrow terminal adjustment `viewportWidth = pt.width - 1` 
- Ensure responsive design still functions correctly for terminals < 60 width
- Verify messageHeight calculations remain correct
- Content area utilizes full width available from tab manager

**Dependencies**: Task 1

### Task 3: Update Input Area Rendering for Borderless Design
**Acceptance Criteria**:
- Review `renderInputArea()` method to ensure it works without message area borders
- Verify container width calculations in input area use full available space
- Ensure input area padding/margins provide appropriate visual spacing
- Maintain responsive adjustments for narrow terminals
- Input area renders without unnecessary border styling

**Dependencies**: Task 1, Task 2

### Task 4: Update or Remove Unused Border Style Methods
**Acceptance Criteria**:
- Review `GetPlanningBorderStyle()` method in styles.go for continued usage
- Remove method entirely if no longer used, or update documentation if still needed for other components
- Ensure no other planning tab components depend on this border style
- Clean up unused style definitions if applicable

**Dependencies**: Task 1, Task 2, Task 3

### Task 5: Verify Layout Consistency and Functionality
**Acceptance Criteria**:
- All existing planning tab functionality works correctly (messaging, streaming, navigation)
- Layout remains consistent with Issue #195 two-row footer specification
- Footer system continues to provide visual separation at bottom
- Content displays correctly across terminal widths (narrow, medium, wide)
- No regressions in responsive design behavior
- Message display, input handling, and scrolling work as expected

**Dependencies**: Task 1, Task 2, Task 3, Task 4

## Implementation Notes

### Key Changes Overview:
1. **Remove Border Container**: Direct rendering of viewport content without border wrapper
2. **Full Width Usage**: Update all width calculations to use available space completely  
3. **Maintain Functionality**: Preserve all existing features while removing visual borders
4. **Footer Integration**: Rely on existing footer system for visual separation

### Responsive Design Preservation:
- Narrow terminal adjustments still apply to content formatting
- Padding and margin calculations updated for borderless design
- Input area responsive behavior maintained
- Message styling and spacing preserved

### Architecture Benefits:
- Maximizes content area for conversation display
- Eliminates redundant visual boundaries
- Maintains clean separation through footer system
- Reduces visual noise and improves readability
- Consistent with Issue #195 layout specification

This design removes visual complexity while preserving all functionality, resulting in a cleaner interface that maximizes space utilization for the primary use case: planning conversations.
