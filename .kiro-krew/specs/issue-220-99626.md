# Improve Cursor Visibility Management in Planning Tab

**Issue:** #220 - Improve cursor visibility management in planning tab  
**Closes:** #220

## Problem Analysis

### Root Cause: Dual Cursor Visibility Issue

The planning tab currently allows both the message input area (`textinput.Model` at line 96 in `planning_tab.go`) and the footer command input (`AutocompleteInput` wrapper) to display cursors simultaneously, creating visual confusion about which input has focus.

**Current Focus Management:**
- **Planning Tab Input**: Uses `focusInput` boolean flag (line 43) with Tab key handling (lines 607-614)
- **Footer Input**: Uses `AutocompleteInput.textinput.Focus()`/`Blur()` methods
- **Problem**: No coordination between these two input systems

**Observed Behavior:**
1. Both inputs may show cursors simultaneously
2. User cannot easily determine which input will receive keystrokes
3. No solid cursor enforcement (current cursors may blink by default)
4. Focus state is not properly preserved between tab switches

### Current Architecture Analysis

**Planning Tab Focus Management (lines 607-614):**
```go
case "tab":
    // Toggle focus between viewport and input
    pt.focusInput = !pt.focusInput
    if pt.focusInput {
        cmds = append(cmds, pt.textinput.Focus())
    } else {
        pt.textinput.Blur()
    }
```

**Footer Input Focus Management:**
- Controlled globally in main TUI controller
- Uses `input.Focus()`/`input.Blur()` methods from `AutocompleteInput`
- No coordination with planning tab internal focus

## Solution Approach

Implement a **coordinated dual-input cursor management system** that ensures single cursor visibility through proper focus coordination between planning tab message input and footer command input.

### Core Strategy

1. **Focus Coordination Protocol**: Establish clear communication between planning tab and footer input systems
2. **Solid Cursor Enforcement**: Configure both input systems to use solid (non-blinking) cursors 
3. **Exclusive Focus Management**: Ensure only one input shows a cursor at any time
4. **State Preservation**: Maintain focus state across tab switches

### Technical Approach

**Cursor Mode Configuration:**
- Configure `textinput.Model` instances to use solid cursor mode
- Implement cursor visibility coordination protocol

**Focus State Management:**
- Extend planning tab to communicate focus state with parent TUI
- Implement focus handoff mechanism between inputs
- Add default focus assignment (message input on tab open)

## Technical Notes

### Cursor Mode Configuration
The implementation will use Bubble Tea's textinput cursor mode settings to enforce solid cursors instead of the default blinking behavior.

### Focus Coordination Protocol
The solution implements a lightweight protocol where the planning tab can signal focus changes to the parent TUI, enabling coordinated cursor visibility management without tight coupling.

### State Preservation
Focus state preservation will be handled through the existing tab manager infrastructure, extending it to track and restore input focus states across tab switches.

### Backward Compatibility
This solution maintains all existing keyboard shortcuts and functionality while adding the coordinated cursor management system.
