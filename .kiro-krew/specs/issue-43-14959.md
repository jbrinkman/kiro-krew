# Theme Command Implementation Design Specification

**Closes #43**

## Solution Approach

Add a `theme` command to the REPL that allows users to view the current theme and switch between available themes. The solution leverages existing theme loading infrastructure in `internal/config/themes.go` and integrates seamlessly with the current REPL command system in `internal/tui/tui.go`.

### Key Architectural Decisions

1. **Extend Existing Command System**: Add theme command handling to the existing `executeCommand()` function in `tui.go`
2. **Leverage Current Theme Infrastructure**: Use existing `LoadTheme()` function and `validThemeNamePattern` for validation
3. **Add Configuration Persistence**: Implement config saving functionality to preserve theme changes
4. **Live Theme Switching**: Update model styles and trigger screen refresh for immediate visual feedback
5. **Robust Error Handling**: Provide helpful error messages with available theme listings

## Implementation Notes

### Configuration Preservation
- Must preserve existing config comments and structure
- Only update the `theme` field value
- Handle cases where theme field doesn't exist (add it)
- Handle cases where theme field is commented out (uncomment and update)

### Theme Name Validation
- Use existing `validThemeNamePattern` from `themes.go`
- Validate theme file exists before switching
- Provide helpful error messages with available options

### UI Update Strategy
- Update model's `styles` field with new theme
- Use `tea.ClearScreen` to force immediate refresh
- Ensure smooth transition without flicker

### Error Recovery
- Fall back to current theme if switching fails
- Provide clear error messages for all failure cases
- Never leave the application in an inconsistent state

This specification provides a complete roadmap for implementing the theme command feature while leveraging existing infrastructure and maintaining code quality standards.
