# Design Specification: Implement Missing Deterministic Checkers for Evaluation Rubrics

**Issue**: #73  
**Closes**: #73

## Problem Statement

The evaluation system has deterministic criteria defined in rubrics but lacks checker implementations for several key criteria. The `scoreDeterministic` function in `internal/eval/runner.go` only implements checkers for `completeness`, `file_reference`, and `file_naming` patterns, leaving 4 deterministic criteria unimplemented:

1. `acceptance_criteria_testability` (planner.yaml)
2. `test_execution` (validator.yaml) 
3. `code_correctness` (builder.yaml)
4. `test_coverage` (builder.yaml)

## Solution Approach

### High-Level Strategy

Extend the existing `scoreDeterministic` function in `internal/eval/runner.go` by adding 4 new checker implementations that follow the established pattern-matching approach used by existing checkers.

### Architectural Decisions

1. **Pattern-Based Detection**: Use string pattern matching similar to existing checkers to identify criterion types
2. **Heuristic Analysis**: Implement practical heuristics that can reliably detect the presence of expected elements
3. **Conservative Scoring**: Prefer false negatives over false positives to maintain evaluation integrity
4. **Consistent Interface**: Maintain the existing function signature and return format

### Implementation Strategy

Each checker will:
- Parse the test case output for specific indicators
- Apply scoring logic based on detected elements
- Return appropriate scores with descriptive reasoning
- Handle edge cases gracefully with appropriate fallbacks

## Detailed Checker Specifications

### 1. acceptance_criteria_testability Checker

**Purpose**: Verify that acceptance criteria include testable assertions or commands

**Detection Logic**:
- Look for checkbox lists (`- [ ]` patterns) in output
- Count criteria that contain testable verbs (returns, accepts, validates, displays, etc.)
- Check for specific assertion patterns (status codes, expected outputs, error conditions)

**Scoring**:
- Count testable criteria vs total criteria
- Score = (testable_count * max_score) / total_count
- Minimum score of 1 if any testable criteria found

### 2. test_execution Checker

**Purpose**: Verify that validation commands were actually executed and results reported

**Detection Logic**:
- Look for command execution indicators (`$`, `go build`, `go test`, exit codes)
- Search for execution result patterns (✓, ✗, "exit code", "passed", "failed")
- Count commands that show actual execution vs just mentions

**Scoring**:
- Score based on ratio of executed commands to mentioned commands
- Bonus points for showing exit codes and detailed results
- Full score if multiple command types executed with results

### 3. code_correctness Checker

**Purpose**: Verify that generated code compiles/runs without errors

**Detection Logic**:
- Look for build success indicators ("Build passes", "✓", "exit code 0")
- Search for compilation/syntax validation evidence
- Check for error-free execution mentions

**Scoring**:
- Full score if clear build/compile success indicators found
- Partial score for syntax validation without errors
- Zero score if build errors mentioned or no validation evidence

### 4. test_coverage Checker

**Purpose**: Verify that appropriate tests are included for new functionality

**Detection Logic**:
- Look for test-related content (test files, test functions, test execution)
- Search for testing framework usage (go test, jest, pytest patterns)
- Check for test description or test case mentions

**Scoring**:
- Score based on evidence of test creation or execution
- Higher scores for multiple test types or comprehensive coverage
- Minimum score for any test-related activity
